1. EXECUTIVE SUMMARY
================================================================================

This plan delivers a working SOCMINT Suspect Profiling MVP in 4 weeks.
Version 2.0 integrates the full OSINT Cabal Linux CLI toolkit, expanding
the tool suite from 6 to 35+ active tools across all pipeline stages.

Given a single seed (username OR email OR phone), the system will:

  DISCOVER     — Automatically find linked accounts across 1000+ platforms
                 using 15+ integrated OSINT tools running in parallel tiers

  CORRELATE    — Score identity links probabilistically using weighted signals
                 with a minimum 2-signal corroboration rule, now enriched
                 with Google account data, Gravatar correlation, WhatsApp
                 linkage, bio similarity, and breach credential reuse

  PRESERVE     — Screenshot and hash every positive hit at discovery time
                 so evidence cannot be lost to platform deletions, backed up
                 to Wayback Machine and archived via WayBackPack

  VISUALISE    — Display a live identity graph showing linked accounts,
                 confidence scores, signal breakdowns, and platform clusters

  REPORT       — Generate a structured JSON + PDF evidence package with
                 full provenance, audit trail, and SHA-256 signed bundle

  PLATFORM DIG — Run platform-specific tools (TikTok, Snapchat, Telegram,
                 Mastodon, Bluesky, 4chan, ProtonMail) on confirmed hits
                 for deeper profile data beyond basic account discovery

  DORK & CRAWL — Passive exposure discovery via Google dorking and web
                 crawling without touching target platforms directly

The system is built entirely on free, open-source tools and runs locally
via Docker Compose. No paid APIs, no cloud spend, no licensing costs.

This is demonstrably superior to manual investigation:
  Manual: 1 analyst, 20+ platforms, hours of work, no audit trail
  This system: Automated, parallel, scored, preserved, auditable, < 60 min
               35+ tools, 1000+ platforms, 10+ signal types, full chain

5. MODULE BREAKDOWN
================================================================================

MODULE 0: EVIDENCE SCHEMA & PROVENANCE SERVICE
  Build this first. Everything else produces or consumes this schema.

  EvidenceUnit:
    evidence_id          : UUID (auto-generated)
    case_id              : str  (from Legal Gate)
    run_id               : str  (from Seed Manager)
    tool_name            : str  (e.g. "sherlock", "ghunt", "tiktok_userdata")
    tool_version         : str
    tool_tier            : int  (1=fast sweep, 2=deep, 3=passive, 4=triggered)
    source_platform      : str  (e.g. "twitter", "github", "tiktok")
    source_tier          : int  (1=API, 2=public web, 3=archive, 4=inferred)
    seed_type            : str  ("username" | "email" | "phone")
    seed_value           : str
    result_type          : str  ("account_found" | "email_registered" |
                                 "breach_hit" | "gravatar_hit" | "google_hit" |
                                 "whatsapp_hit" | "domain_hit" | "dork_hit" |
                                 "archive_hit" | "unavailable" | "blocked")
    result_value         : str  (URL, username, email, etc.)
    confidence_raw       : float
    signal_weights       : dict (breakdown of contributing signals)
    timestamp_collected  : datetime (UTC)
    timestamp_preserved  : datetime (UTC)
    snapshot_ref         : str  (MinIO object path)
    snapshot_hash        : str  (SHA-256 of preserved artifact)
    wayback_ref          : str  (Wayback Machine archive URL if saved)
    platform_enrichment  : dict (Tier 4 tool output, if run)
    analyst_id           : str
    notes                : str

  ProvenanceService methods:
    create_evidence_unit(data)     → EvidenceUnit
    validate_schema(unit)          → bool
    write_to_db(unit)              → UUID
    compute_hash(artifact_bytes)   → str
    log_audit_event(type, actor)   → None
    attach_enrichment(id, data)    → None


MODULE 1: LEGAL GATE & SEED MANAGER
  Unchanged from v1.0. Validates mandatory fields before any tool fires.
  Normalises the seed input. Issues case_id and run_id.
  Writes intake record to audit log.

  Required fields (analyst must provide all):
    authority_id         : Authorisation reference number
    agency_id            : Agency/organisation identifier
    analyst_id           : Analyst identifier
    supervisor_approval  : Boolean — supervisor signed off
    purpose_statement    : Free text — why collection is authorised
    target_category      : "cybercrime" | "fraud" | "harassment" | "research"
    jurisdiction         : Country/region code
    retention_period     : Days
    seed_type            : "username" | "email" | "phone"
    seed_value           : The actual seed

  Normalisation rules:
    Username: lowercase, strip @prefix, strip whitespace
    Email:    lowercase, strip whitespace, validate format
    Phone:    E.164 format normalisation

MODULE 3: TOOL EXECUTOR (v2.0)
  Async parallel execution in 4 tiers.
  Tier 1 fires first; Tier 2 fires in parallel with Tier 1.
  Tier 3 fires in background from case start (non-blocking).
  Tier 4 fires only when a confirmed platform hit arrives.

  Celery task groups:
    group_tier1  = group(run_tier1_tools.s(seed))
    group_tier2  = group(run_tier2_tools.s(seed))
    group_tier3  = group(run_passive_tools.s(seed))
    chord([group_tier1, group_tier2], aggregate_results.s())

  Rate limit config (per tool, config.yaml):
    sherlock:        cool_down_seconds: 5
    maigret:         cool_down_seconds: 5
    holehe:          cool_down_seconds: 10
    h8mail:          cool_down_seconds: 5
    ghunt:           cool_down_seconds: 15  (Google rate-sensitive)
    whatsMyName:     cool_down_seconds: 5
    nexfil:          cool_down_seconds: 5
    social_analyzer: cool_down_seconds: 8
    dorks_eye:       cool_down_seconds: 30  (Google — use Tor)
    dorksint:        cool_down_seconds: 30  (Google — use Tor)
    toutatis:        cool_down_seconds: 20  (Instagram — rate-sensitive)
    telegram_tools:  cool_down_seconds: 10

  Execution audit fields (same as v1.0):
    run_id, tool_name, tool_tier, seed_type, seed_value,
    started_at, completed_at, status, result_count, error_message


MODULE 4: PRESERVATION SERVICE (v2.0)
  On every positive hit:
    1. Fetch full HTML of the profile URL (httpx)
    2. Store raw HTML to MinIO: cases/{case_id}/{evidence_id}/raw.html
    3. Compute SHA-256 hash of stored artifact
    4. Take GoWitness screenshot if URL is web-accessible
    5. Store screenshot to MinIO: cases/{case_id}/{evidence_id}/screenshot.png
    6. Submit Wayback Machine save request (free CDX API)
    7. WayBackPack: download archived version if URL has prior snapshots
    8. Return snapshot_ref, snapshot_hash, and wayback_ref to executor

  IMPORTANT: Preservation runs BEFORE EvidenceUnit is written to DB.
  If preservation fails: log warning, continue (don't drop the evidence).

  MinIO bucket structure (v2.0):

9. CORRELATION ENGINE (UPGRADED v2.0)
================================================================================

9.1 SCORING FORMULA
  confidence_score = SUM(signal_weight * decay_factor)
                   - SUM(conflict_penalties)
                   [+ semantic_bonus if pgvector enabled: 0-10]
                   [+ gravatar_bonus if Hashtray confirms: 0-8]
                   [+ google_bonus if Ghunt confirms: 0-10]
                   [+ whatsapp_bonus if Email2WhatsApp confirms: 0-7]

9.2 POSITIVE SIGNAL WEIGHTS
+------------------------------------------+--------+------------------------+
| Signal                                   | Weight | Implementation         |
+------------------------------------------+--------+------------------------+
| Identical username (exact, normalised)   |   25   | string equality after  |
|                                          |        | lowercasing + leet fix |
+------------------------------------------+--------+------------------------+
| Same email or direct variation           |   20   | exact match or +alias  |
+------------------------------------------+--------+------------------------+
| Breach credential reuse (H8mail)         |   18   | * decay applied        |
+------------------------------------------+--------+------------------------+
| Identical profile photo (pHash <= 8)     |   15   | imagehash pHash        |
+------------------------------------------+--------+------------------------+
| Ghunt Google account confirmed           |   12   | Google profile match   |
+------------------------------------------+--------+------------------------+
| Bio/description similarity >= 80%        |   10   | sentence-transformers  |
|                                          |        | cosine similarity      |
+------------------------------------------+--------+------------------------+
| Overlapping linked external URLs         |   10   | URL set intersection   |
+------------------------------------------+--------+------------------------+
| Gravatar profile confirmed (Hashtray)    |    8   | email/username → avatar|
+------------------------------------------+--------+------------------------+
| WhatsApp linkage confirmed               |    7   | Email2WhatsApp probe   |
+------------------------------------------+--------+------------------------+
| Username edit distance <= 2 chars        |    5   | Levenshtein distance   |
+------------------------------------------+--------+------------------------+
| Dork hit confirms name + platform        |    5   | Google dork result     |
+------------------------------------------+--------+------------------------+
| Platform enrichment metadata match       |    5   | Tier 4 tool output     |
|  (bio, location, join date overlap)      |        |                        |
+------------------------------------------+--------+------------------------+

9.3 CONFLICT PENALTIES
+------------------------------------------+--------+------------------------+
| Signal                                   | Penalty| Implementation         |
+------------------------------------------+--------+------------------------+
| Account identified as bot/automated      |  -15   | Pattern detection      |
+------------------------------------------+--------+------------------------+
| Timezone mismatch > 4hr (systematic)     |  -10   | Posting time analysis  |
+------------------------------------------+--------+------------------------+
| Primary language mismatch                |   -8   | langdetect on bio/post |
+------------------------------------------+--------+------------------------+
| Username on generic blacklist            |   -8   | admin, user, test, etc.|
+------------------------------------------+--------+------------------------+
| Account creation date impossible         |   -5   | Temporal logic check   |
+------------------------------------------+--------+------------------------+
| Platform allows duplicate usernames      |   -5   | Config flag per        |
|                                          |        | platform               |
+------------------------------------------+--------+------------------------+

9.4 CONFIDENCE THRESHOLDS
  75 – 100  =  HIGH      Analyst confirmation required before report
  50 –  74  =  MEDIUM    Include with caveat, review recommended
  25 –  49  =  LOW       Background context only
   0 –  24  =  DISCARD   Insufficient evidence, raw log only

  MINIMUM CORROBORATION RULE (non-negotiable):
  No identity link is written to Neo4j from a single signal alone,
  regardless of how high the weight is.
  Minimum 2 independent signals required.

9.5 EVIDENCE DECAY FORMULA
  decay_factor = max(MIN_DECAY, 1.0 - (evidence_age_days / DECAY_PERIOD))

  +----------------------------+-------------+------------+
  | Signal Type                | Decay Period| Min Decay  |
  +----------------------------+-------------+------------+
  | Breach credential reuse    | 1095 (3yr)  |   0.20     |
  | Email on platform          |  730 (2yr)  |   0.30     |
  | Username match             |  365 (1yr)  |   0.50     |
  | Profile photo match        |  180 (6mo)  |   0.60     |
  | Bio/description text       |   90 (3mo)  |   0.70     |
  | Dork hit                   |   30 (1mo)  |   0.50     |
  +----------------------------+-------------+------------+

  Signals NOT subject to decay (durable identifiers):
    Identical username (exact), identical email, URL overlap,
    Google account confirmation, Gravatar confirmation

9.6 LEET-SPEAK NORMALISATION (for username matching)
  Apply before any username comparison:
    0 → o,  1 → i (or l),  3 → e,  4 → a,
    5 → s,  7 → t,  @ → a

  COMMON USERNAME BLACKLIST (automatic -8 penalty):
    admin, user, test, info, help, support, contact, hello,
    noreply, no-reply, webmaster, postmaster, root, guest

    snapshot_hash        TEXT,
    wayback_ref          TEXT,                   -- NEW

    analyst_id           TEXT NOT NULL,
    notes                TEXT,

16. TECHNICAL RISKS & MITIGATIONS
================================================================================

RISK 1: OSINT tools break during demo
  Likelihood: HIGH (Sherlock, Holehe break as platforms change)
  Mitigation: Run all tools against demo case on Day 24 (Week 4).
              FallbackChainManager auto-skips broken tools.
              Pre-cache 1 full run in PostgreSQL as demo fallback.
              If a tool breaks: frame it — "System detects tool
              failure and auto-routes to fallback chain."

RISK 2: Platform blocks during live demo
  Likelihood: MEDIUM
  Mitigation: Use Tor for rate-sensitive tools during demo if needed.
              Pre-run results cached. Frame blocking as UNAVAILABLE
              (not "not found") — show judges the system distinguishes.

RISK 3: Docker Compose fails on demo machine
  Likelihood: LOW but catastrophic
  Mitigation: Test on judge's machine OS one week before demo.
              Have backup laptop with stack pre-running.
              Screenshot walkthrough as last resort.

RISK 4: Go binary compilation issues on demo machine
  Likelihood: MEDIUM (Go version mismatch)
  Mitigation: Pre-compile all Go binaries for linux/amd64.
              Store compiled binaries in /tools/go/ (not just source).
              Document exact Go version in README.

RISK 5: Neo4j graph rendering slow with many nodes
  Likelihood: MEDIUM
  Mitigation: Limit graph display to top 50 nodes by confidence.
              Use NetworkX + Plotly as fallback if Neo4j Browser lags.

RISK 6: Ghunt Google cookie expiry
  Likelihood: MEDIUM (Google cookies expire)
  Mitigation: Re-authenticate Ghunt cookies 48 hours before demo.
              If Ghunt fails during demo, show cached enrichment result.
              Label clearly as "Google account enrichment — cached result."

RISK 7: Holehe false positives (password reset probing not 100% reliable)
  Likelihood: MEDIUM
  Mitigation: Require 2-signal corroboration (the rule handles this).
              Label Holehe results as Tier 2 (public web, unverified).
              Holehe hit alone never produces an identity link.

RISK 8: Tier 4 platform tools requiring session tokens (Instagram, Telegram)
  Likelihood: HIGH (session tokens are fragile)
  Mitigation: Set up fresh session tokens on Day 24.
              If a platform tool fails: frame as limitation, show
              what the tool would return from a pre-run demo cache.
              Do not demo live Toutatis unless token confirmed working.

RISK 9: Team integration conflicts (merge hell in Week 3)
  Likelihood: MEDIUM
  Mitigation: API contracts defined in Week 1 (FastAPI endpoint specs).
              Each person owns their module's endpoint — others call it.
              Daily 15-minute standups from Day 8 onward.
              Feature freeze Day 28 — no new code after that.

RISK 10: Dorking tools triggering Google CAPTCHA
  Likelihood: HIGH if not Tor-routed
  Mitigation: ALL dorking tools MUST use Tor (GoTOR).
              Max 10 queries per case, 30s cooldown between queries.
              If CAPTCHA hit during demo: skip dorking, show cached
              result. Frame as "Passive recon — runs in background,
              no platform contact required for core demo."
